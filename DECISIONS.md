**정찰 결과**: 청약홈 분양정보는 한국부동산원 **data.go.kr OpenAPI(REST·JSON)** 로 구조화 제공됨(서비스 `15098547` 분양정보 / `15098905` 경쟁률·특공). 엔드포인트 `https://api.odcloud.kr/api/ApplyhomeInfoDetailSvc/v1/{getAPTLttotPblancDetail,getAPTLttotPblancMdl,...}` 실호출로 유효 확인(미신청이라 -4, 경로는 정상). **분양가·주택형·세대수·일정·지역우선·URL이 필드로** 나옴 → PDF→Sonnet 추출 불필요(임대보다 쉬움). 상세 = `청약홈_분양_API_노트.md`.
**결정**: 분양은 청약홈 OpenAPI를 소스로 채택. **가점 매칭까지 GO.** 1차는 API만(추출 0). 갭(가점/추첨 비율·전매제한·실거주의무)만 필요 시 공고문 PDF로 2차 보강(applyhome 접근 가능 확인).
**활용신청 승인·실데이터 확인(2026-06-22, HTTP 200)**: `15098547` 동작. 실측 추가 — Mdl이 분양가(`LTTOT_TOP_AMOUNT` 만원)뿐 아니라 **특공 세부배분(다자녀·신혼·생초·노부모·기관추천 등) 전부 필드로** 제공(예상보다 좋음, 갭 축소). 조인키 `PBLANC_NO==HOUSE_MANAGE_NO`, `PBLANC_URL`로 공고 직링크. ⚠️ `cond` 기간필터 미동작 → 최신순 정렬 활용 클라 측 기간컷. 다음 액션 = `applyhome-collect.mjs` 구현.
**이유**: 한 소스로 공공+민영 분양 동시. 구조화 JSON이라 no-API 규칙과 정합(추출 단계 자체가 사라짐). 이전 결정(2026-06-22 "분양은 청약홈 소스 확보 후")의 후속 — 소스 확보 경로가 스크래핑이 아니라 OpenAPI로 확정됨.

## 2026-06-23 · 신혼희망타운(공공분양) 표준 자격 인코딩 — "확인필요" 대신 단정
**문제**: 공공분양을 일괄 "소득·자산 확인필요"로 punt → 신혼희망타운은 자격 기준이 표준 정책값으로 명확한데도 단정 못 함(사용자 지적).
**결정**: `match-core`에 신혼희망타운(공고명 `/신혼희망타운/`) 전용 게이트 `evalNHT` — 대상계층(혼인7년이내·만7세미만자녀·예비신혼) + 무주택 + 청약통장(6개월·6회) + 총자산(3.62억, 2026적용) + 소득(도시근로자 130%/맞벌이 140%, 2024적용 표). 통과시 **지원가능**, 명확한 미달은 **지원불가**로 단정. 정책값은 **연도·출처 라벨**을 메시지에 명시(드리프트 대비). 진짜 불확실한 "**본청약=사전청약 당첨자 우선, 신규 일반공급(60%)/추첨(10%) 물량 여부**"만 참고로 분리(판정 안 깎음).
**이유**: 표준화된 기준은 결정론으로 단정하는 게 정직성에 부합(추측이 아니라 공시된 정책값). 연도 라벨로 "공고문과 다를 수 있음" 고지. 출처: LH/마이홈/뉴홈. 잔여 공공분양(신혼희망타운 외)은 일반공급 소득·자산표 미인코딩이라 여전히 확인필요(추후 확장 여지).

## 2026-06-23 · 매칭 로직 단일소스 `match-core.mjs` + 브라우저 실시간 재계산 + 동작 URL 검증
**결정**: 매칭 로직을 순수모듈 `match-core.mjs`(`createMatcher(P,today).evaluate(req)`)로 추출 → `match.mjs`(node·IO/CLI)와 웹페이지가 **공유**(단일 소스). `build-site.mjs`는 requirements+프로필+`match-core`(export 제거)를 `site/index.html`에 인라인 → 브라우저가 직접 매칭. **프로필 편집 패널(드로어)** 에서 조건 바꾸면 즉시 재계산(localStorage 저장). 자체완결 HTML(서버 불필요).
**URL 검증**: LH 상세페이지 `selectWrtancInfo.do`는 GET 시 "비정상 경로" 에러(POST 전용) → 원문 링크를 **공고문 PDF `lhFile.do`(GET 검증됨)** 로 교체(`pickLink`). 분양(청약홈) 상세는 GET 정상. 링크 접근성 curl 검증 후 반영.
**이유**: 매칭이 두 벌이면 드리프트(검증 원칙 위반) → 코어 단일화. 정적 페이지는 node 불가라 조건 수정=클라 매칭이 유일 → 코어 인라인. 깨진 URL 방치는 신뢰도 직결 → 출처별 동작 URL 검증·교체. 신뢰감: 출처·기준일·"확인필요/참고/추정" 정직표기 UI 연장.

## 2026-06-23 · 분양 가점계산기 = match.mjs 분양 분기 (가점 84점, 갭은 판정 안 깎음)
**결정**: `match.mjs`가 `derived/lh`+`derived/applyhome` 둘 다 읽고 `상품구조==="분양"`이면 `evaluateSale`. 분양 판정 게이트 = **무주택+청약통장**(+공공분양은 소득·자산컷=확인필요). 민영 일반공급은 무주택+통장이면 **지원가능**(추첨모수 보장), 가점 84점〔무주택32+부양35+통장17〕은 당첨가능성으로 별도 표시. 공공분양은 순차제(저축총액·납입). 청약순위(통장기간×규제임계)·지역우선 tier(거주지×공고)·특공해당(신혼/다자녀/신생아 결정론) 계산.
**이유**: 가점/추첨비율·전매제한·실거주의무는 **자격이 아니라 당첨후 조건** → "참고"로 분리(판정 안 깎음). 특공 해당도 일반공급 외 추가기회라 판정 유지+참고. 안 그러면 특공 해당자가 오히려 ⚠️로 강등되는 역전 발생(실측 발견·수정). 정직성 원칙(추측 금지) 유지하되 일반공급 지원가능은 확정으로.

## 2026-06-22 · 공급형태(지원형 vs 실물) 결정론 분류
"전세보증금만 지원(집은 본인 탐색)" 정책과 "실물 호실 공급"은 사용자에게 완전히 다른데 유형명만으론 안 갈림.

**이유**: 든든전세처럼 "전세형이지만 실물 목록 있는" 경계도 주택목록 유무로 정확히 갈림. 기존 필드만으로 결정론 분류 가능.
